1.1 Vision & Narrative
------------------------
Mozart Token is a cultural-tech initiative that preserves and celebrates Austria's
musical legacy on the blockchain. Our mission is to bridge classical heritage with
decentralized technology, creating a token ecosystem that supports music education,
digital art, and community-driven cultural preservation.

The project is built on a simple yet powerful narrative: "Preserving Austria's
musical legacy on-chain." We are not building a speculative meme token. Mozart Token
is designed as a utility-driven cultural asset with real-world applications in music
NFTs, event ticketing, education grants, and DAO governance.

Core Theme: Culture × Blockchain × Community × Education

1.2 Why Mozart
--------------
Wolfgang Amadeus Mozart (1756–1791) is the most famous Austrian musician in history
and a universal symbol of genius, culture, and timeless artistry. As Mozart died in
1791, his name and likeness are in the public domain, making this branding legally
safe for commercial use without licensing restrictions.

Mozart represents values that align perfectly with blockchain ethos:
  • Decentralization of art (his music belongs to the world)
  • Timelessness (his work transcends eras)
  • Global recognition (universal brand awareness)

This is a legitimate cultural project, not a hype-driven scheme.

2.2 Smart Contract Specifications
---------------------------------
We require a production-ready ERC-20 smart contract with the following features:

Core Contract Features
  • Fixed Total Supply: 1,000,000,000 MOZART minted at deployment
  • No Minting Function: Supply is immutable after launch
  • Burn Mechanism: Users can burn tokens permanently (deflationary feature)
  • Pausable: Emergency pause functionality (multi-sig controlled)
  • Blacklist: Ability to restrict malicious addresses
  • Ownable / Access Control: Role-based permission system
  • Fee System: Configurable transaction fee (max 3%) for treasury

Vesting & Allocation Contracts
  • Team Vesting: 24-month linear vesting with 6-month cliff
  • Marketing Vesting: 12-month linear vesting
  • Development Vesting: 18-month linear vesting
  • Liquidity Lock: Minimum 12-month liquidity lock mechanism

The vesting contracts must support:
  • Multiple beneficiary addresses per allocation category
  • Revocable grants (for team/marketing allocations)
  • On-chain claim functionality for beneficiaries
  • Real-time view of vested vs. unvested amounts

2.3 Security & Audit Requirements
-----------------------------------
Security is our highest priority. The following standards must be met:

  • ReentrancyGuard: All external calls protected against reentrancy attacks
  • SafeMath: Overflow/underflow protection (Solidity 0.8.x built-in + explicit checks)
  • Slither Analysis: All Slither detectors must pass (no high/medium severity issues)
  • Unit Test Coverage: Minimum 95% code coverage using Hardhat + Ethers.js
  • Formal Audit: Contract must pass review by a reputable audit firm (CertiK, Hacken,
    or equivalent)

Deployment Configuration
  • Deployment via multi-sig wallet (Gnosis Safe) — 3-of-5 signers
  • Verified and published source code on Etherscan / Polygonscan
  • Mainnet deployment only after successful testnet deployment + 2-week observation
